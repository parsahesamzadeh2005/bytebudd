### Key Backend Modules Explained

#### `core/config.py`
Loads all config from environment variables using Pydantic Settings. Single source of truth. Import `settings` anywhere.

#### `core/security.py`
Two functions you'll use: `hash_password()` and `verify_password()` for bcrypt. `create_access_token()` for JWT. Stateless — no session storage.

#### `core/encryption.py`
Database passwords are encrypted with Fernet (AES) before storing in Postgres. The `ENCRYPTION_KEY` env var is the master key. Always decrypt before connecting to target DBs.

#### `services/sql_guard.py` ⭐ CRITICAL
This is the security core. Every SQL generated by the LLM passes through here before execution. It:
1. Strips markdown formatting from LLM output
2. Blocks dangerous keywords (INSERT, UPDATE, DELETE, DROP, etc.)
3. Parses SQL with sqlglot and verifies it's a SELECT
4. Adds LIMIT 1000 if missing, caps at 1000 if higher

#### `services/query_pipeline.py` ⭐ CORE FLOW
The main pipeline that ties everything together:
1. Load DBConnection from database
2. Call `get_connector()` to get the right driver
3. Call `connector.get_schema()` to read table structure
4. Call `build_sql_prompt()` to create the LLM prompt
5. Call `ollama_client.generate()` to get SQL from AI
6. Call `validate_sql()` to check safety
7. Call `connector.execute_query()` to run the query
8. Yield SSE events at each step

#### `db_connectors/`
Each connector implements the same 3-method interface (`BaseConnector`):
- `test_connection()` → connection health check
- `get_schema()` → returns a text description of all tables/columns
- `execute_query(sql)` → runs the query, returns list of dicts

### Key Frontend Files Explained

#### `lib/api.ts`
All network requests live here. The `streamQuery()` function is the most important — it POSTs to the backend and reads the SSE stream using the Fetch API's `ReadableStream`.

#### `components/chat/ChatWindow.tsx`
Manages the chat state. On submit:
1. Adds user message to state
2. Adds empty assistant placeholder (shows thinking dots)
3. Calls `streamQuery()` which parses each SSE event
4. Updates the placeholder message as events arrive

#### `components/chat/DataGrid.tsx`
Displays query results as a scrollable table with client-side pagination (20 rows per page). Shows a warning if 1000-row limit was hit.

## Database Layer

### ByteBudd's Internal Database (Postgres)

Stores ByteBudd's own data — users, connections, conversations.

```
users
  id, email, password_hash, role, is_active, created_at

db_connections
  id, user_id, name, db_type, host, port, database_name,
  username, encrypted_password, sqlite_path, is_active, created_at

conversations
  id, user_id, db_connection_id, title, created_at, updated_at

messages
  id, conversation_id, role, content, generated_sql, created_at

audit_logs
  id, user_id, db_connection_id, question, generated_sql,
  execution_time_ms, row_count, success, error_message, created_at
```

### Target Databases (User's Data)

The databases users connect to. ByteBudd connects with a fresh connection per query. No persistent pool is kept — keeps it simple and avoids connection leaks.

## Data Flow (End to End)

```
1. User types: "Show me the last 10 orders"

2. ChatWindow.tsx calls streamQuery()
   POST /api/v1/query/stream
   Body: { question, conversation_id, db_connection_id }

3. query.py validates conversation ownership

4. query_pipeline.py starts:

   a. Loads DBConnection from Postgres
   b. Calls get_connector(db_conn) → MySQLConnector (for example)
   c. Calls connector.get_schema() → "TABLE wp_posts (...)\nTABLE wp_users (...)"
   d. Calls build_sql_prompt(question, schema) → full LLM prompt
   e. Calls ollama_client.generate(prompt) → "SELECT * FROM wp_posts WHERE post_type='shop_order' ORDER BY post_date DESC LIMIT 10"
   f. Calls validate_sql(raw_sql) → verified safe SQL
   g. Calls connector.execute_query(safe_sql) → [{id:1, ...}, {id:2, ...}]

5. SSE events yielded:
   event: thinking → "Reading schema..."
   event: thinking → "Generating SQL..."
   event: sql      → { sql: "SELECT ..." }
   event: results  → { columns: [...], rows: [...], row_count: 10 }
   event: explanation → { text: "Found 10 rows..." }
   event: done     → {}

6. Frontend receives events, updates UI progressively

7. Messages saved to Postgres
8. AuditLog entry created
```

## How to Extend ByteBudd Safely

### Add a new database type (e.g., MSSQL)
1. Create `backend/app/db_connectors/mssql_connector.py`
2. Implement `BaseConnector` (3 methods)
3. Add `"mssql"` to the Enum in `db_connection.py` model
4. Add case in `connector_factory.py`
5. Add `"mssql"` option in the frontend `databases/page.tsx` form

### Add a new API route
1. Create `backend/app/api/v1/your_feature.py`
2. Add router import in `backend/app/api/v1/__init__.py`
3. Add Pydantic schemas in `backend/app/schemas/`

### Add a new database migration
```bash
docker compose exec backend alembic revision --autogenerate -m "add_column_x"
docker compose exec backend alembic upgrade head
```

### Change the Ollama model
```bash
# In .env:
OLLAMA_MODEL=codellama:13b

# Pull the new model:
docker compose exec ollama ollama pull codellama:13b

# Restart backend:
docker compose restart backend
```
